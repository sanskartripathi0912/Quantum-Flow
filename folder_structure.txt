--------------------------------------------------------------------------------
2. WORKING INFRASTRUCTURE
--------------------------------------------------------------------------------

Quantum-Flow is architected with a decoupled structure designed to offload heavy 
computations to background systems.

Client-Side Layer (Frontend):
  - React Flow provides the visual diagram canvas. It tracks nodes (UI boxes) and 
    edges (wires connecting nodes).
  - Zustand manages the client state. Whenever a node is edited, dragged, or runs, 
    Zustand publishes the update to trigger reactive updates in the UI.

API Routing Gateway Layer (Next.js Server):
  - Protected by Clerk Middleware (`src/middleware.ts`). All calls to the server 
    must carry active session credentials.
  - Endpoints (`/api/crop`, `/api/llm`) do not perform media editing or LLM 
    queries directly. Direct processing in API routes risks HTTP timeouts and 
    blocking resources. Instead, they act as controllers delegating tasks.

Task Queue & Worker Layer (Trigger.dev v3):
  - Trigger.dev acts as an external job queue. It listens for tasks triggered by 
    the API gateway.
  - The tasks run on separate Trigger.dev server environments:
    * `llm-task` runs inside a Node environment, calling Google Gemini.
    * `crop-image-task` uses a special build extension (`aptGet(["ffmpeg"])`) to 
       ensure standard system FFmpeg binaries are available in the background 
       environment.

--------------------------------------------------------------------------------
3. PROJECT OPERATION
--------------------------------------------------------------------------------

Workflow Serialization:
  - Auto-Load: On initial mount, `WorkflowCanvas.tsx` issues a `GET` request to 
    `/api/workflows/load`. If files exist from previous sessions, they populate 
    the canvas.
  - Auto-Save: Every node addition, translation, or edge connection triggers a 
    debounced hook. After a `1500ms` window, a `POST` request is sent to 
    `/api/workflows/save` with the current node/edge JSON data.

Real-Time Observability:
  - While tasks run in the background, Zustand tracks the execution status.
  - Custom nodes check the `data.status` field:
    * `"running"` -> Yellow state indicator + spinner text
    * `"success"` -> Green state indicator + results output display
    * `"error"`   -> Red state indicator + warning notification

--------------------------------------------------------------------------------
4. COMPLETE WORKFLOW EXECUTION PIPELINE
--------------------------------------------------------------------------------

When the user hits the "Run" button, the client-side execution engine parses and 
processes the nodes systematically:

Step 1: Cycle Validation (DAG check)
  - The program counts the "in-degree" (incoming connected edges) for all nodes 
    and checks if they form a cycle using a topological validation routine.
  - If a loop is found, execution halts, warning the user that workflows must 
    always flow in a unidirectional path.

Step 2: Level Sorting
  - The engine resolves levels by running a Kahn's topological sort:
    * Level 0: Source nodes with `0` incoming inputs (Text Nodes, Image Nodes).
    * Level 1: Nodes whose inputs depend purely on Level 0 outputs.
    * Level N: Nodes whose inputs depend on Level N-1 outputs.
  - Grouping nodes by level allows nodes inside the same level to execute in parallel, 
    accelerating execution speeds.

Step 3: Level-by-Level Execution Loop
  - The engine progresses through Level 0 to Level N:
    * Before running a node, it crawls the connected edges backward to grab the 
      output values of all source nodes connected to it.
    * Sets node status to `"running"`.
    * Dispatches HTTP requests to the Next.js API endpoints.

Step 4: Asynchronous Processing (Delegation & Polling)
  - The Next.js API endpoint receives the payload and executes:
    `const handle = await tasks.trigger("task-id", payload);`
  - Next.js pauses the endpoint execution thread and polls Trigger.dev:
    `const run = await runs.poll(handle);`
  - In the background, the Trigger.dev worker picks up the job, runs FFmpeg 
    filters or communicates with Gemini APIs, writes the outcome, and completes.
  - The Next.js API receives the completed output from the poll and returns a 
    standard HTTP JSON response to the client.

Step 5: Visual Update & Log Auditing
  - The React Flow node changes status to `"success"`, displaying the returned image 
    or text.
  - The canvas logs this event by pushing the run parameters, timestamp, and results 
    to the Zustand `runs` list.
  - The run immediately appears in the History panel on the Right Sidebar for audit.
